LiquidFeedback Virtual Machine

v0.0.1 (alpha) - 20131130

Pagina Sourceforge del progetto per il download: 
https://sourceforge.net/projects/lqfbvm/

 

 

LQFB_VM vuole essere uno strumento per consentire una rapida esecuzione  ed esplorazione di LiquidFeedback.

Principali caratteristiche:

  • Virtual Machine realizzata con VirtualBox,
  • Sistema Operativo guest installato Debian GNU/Linux,
  • Piattaforma LiquidFeedBack per l’E-Democracy pronta all’uso.

La piattaforma è stata configurata ed è pienamente funzionante nell’ambiente emulato, per le credenziali di accesso, sia a Linux che a LQFB, si deve consultare il Cap. 2. Per permetterne un utilizzo in una rete reale si suggerisce di utilizzare le istruzioni proposte nel Par. 2.1.

Il Cap. 3 ripercorre, sotto forma di un breve tutorial, i passaggi seguiti per ottenere la configurazione inizialmente a disposizione di LQFB.

1) Premessa

Con il termine virtual machine (VM) si indica un software che, attraverso un processo di virtualizzazione, crea un ambiente virtuale che emula il comportamento di una macchina fisica grazie all’assegnazione di risorse hardware (porzioni di disco rigido, RAM, risorse di CPU, etc.). Il vantaggio che si ottiene è quello di poter offrire contemporaneamente ed efficientemente diversi ambienti operativi separati, ciascuno attivabile su effettiva richiesta, senza modificare pesantemente il sistema fisico reale.

Esistono diversi tipi di virtualizzazione, nell’ambito usato dal progetto è stato creato un ambiente (un PC virtuale) dove far girare un sistema operativo, che può essere lo stesso della macchina ospite o meno, destinato all’esplorazione di caratteristiche che non si vogliono o non si possono avere direttamente sulla macchina fisica.

Il progetto LiquidFeedback Virtual Machine (LQFB_VM) ha lo scopo di realizzare un’immagine di un PC virtuale in cui il sistema operativo installato è Linux il quale ospita e permette l’esecuzione e l’esplorazione della piattaforma LiquidFeedback (LQFB) per l’E-democracy.

 

2) La Virtual Machine

L’immagine della VM è stata creata con VirtualBox, un software open source che emula le architetture x86/amd64/Intel64 e supporta Windows, GNU/Linux e Mac OS X come sistemi operativi HOST (piattaforme su cui eseguire la macchina emulata).

Per utilizzare LQFB_VM occorre pertanto effettuare il download e l’installazione di VirtualBox. L’operazione risulta in genere semplice quindi si rimanda ai vari manuali e tutorial che è possibile reperire in rete.

 

La Macchina Emulata Linux

LQFB è disponibile solo per sistemi operativi GNU/Linux e la distribuzione scelta è Debian nella sua versione 7.1.0-i386 con un setup ridotto al minimo per limitare le dimensioni dell’immagine della VM. Ovviamente partendo da questa base ognuno potrà aggiungere le funzionalità che desidera.

Alla macchina emulata Linux è stato assegnato l’hostname

lfvm.test

l’amministratore accede con le credenziali

user:     root
password: lqfb

è stato creato un generico utente per l’utilizzo normale del sistema

2.1) Configurazione di LQFB_VM

Per rendere funzionante la LQFB_VM nella rete reale, oltre che nell’ambiente virtuale, è necessario personalizzarla in 2 aspetti che riguardano

  • la definizione del parametro absolute_base_url, per il corretto funzionamento dell’interfaccia web di LQFB,
  • la configurazione di Exim, per l’invio delle mail.

Purtroppo gli strumenti a disposizione sono poco amichevoli ed il settaggio dei parametri andrà fatto “a mano”. Per una introduzione ai comandi base per operare in una shell di  Linux si consiglia il testo Linux Facile.

 

2.1.1) Setup dell’absolute_base_url in LQFB

L’absolute_base_url dipende dall’indirizzo IP (o eventualmente dal nome) con cui  è possibile raggiungere la VM nella rete emulata e/o reale. Se non correttamente configurato sarà possibile accedere allinterfaccia web di LQFB solo dalla macchina emulata.

Per la gestione della rete in Virtualbox, affinchè la LQFB_VM sia raggiungibile in maniera semplice, è stata scelta la modalità di Bridged Networking. La VM risulta pertanto esposta direttamente alla rete reale a disposizione dell’HOST ed accede al servizio di DHCP per l’assegnazione del IP e dei DNS.

Per un approfondimento della gestione della rete in VirtualBox si faccia riferimento al suo manuale utente o all’articolo “Networking in VirtualBox“. Per la configurazione di un IP statico in una distribuzione Debian si consulti l’articolo “Network Configuration”.

L’indirizzo IP assegnato dinamicanebte dal DHCP è ricavabile con il comando

root@lfvm:~# ifconfig eth0

in risposta al quale

eth0      Link encap:Ethernet  HWaddr 08:00:27:b3:0e:c1
          inet addr:192.168.1.5  Bcast:192.168.1.255  Mask:255.255.255.0
          inet6 addr: fe80::a00:27ff:feb3:ec1/64 Scope:Link
          UP BROADCAST RUNNING MULTICAST  MTU:1500  Metric:1
          RX packets:1762 errors:0 dropped:0 overruns:0 frame:0
          TX packets:248 errors:0 dropped:0 overruns:0 carrier:0
          collisions:0 txqueuelen:1000
          RX bytes:116414 (113.6 KiB)  TX bytes:137545 (134.3 KiB)

si ottiene che l’IP assegnato alla VM nella rete è, nell’esempio, 192.168.1.5. L’absolute_base_url nella configurazione del frontend deve fare riferimento a tale indirizzo IP.

Dalla shell dei comandi dell’utente root con un editor di testi aprire il file di configurazione myconfig.lua

root@lfvm:~# pico /opt/liquid_feedback_frontend/config/myconfig.lua

scorrere fino a raggiungere la riga per la configurazione dell’ absolute_base_url e correggere con  l’IP della VM (nel caso in questione 192.168.1.5)

-- Absolute base url of application
-- ------------------------------------------------------------------------
config.absolute_base_url = "http://192.168.1.5/lf/"

Per una veloce verifica basta raggiungere dalla rete esterna, ad es. con un browser dalla macchina host, l’indirizzo http://192.168.1.5/lf/

2.1.2) Setup di Exim per l’utilizzo di un server SMTP esterno

LQFB_VM utilizza Exim per l’invio delle email agli utenti. La configurazione predisposta per Exim è quella che permette di usare un server SMTP esterno per  l’inoltro della posta elettronica. Se non correttamente configurato sarà possibile l’invio delle mail solo agli utenti locali della VM ( xxx@lfvm.test) e non a quelli remoti.

Dalla shell dei comandi dell’utente root eseguire il seguente comando per ri-configurare Exim4:

root@lfvm:~# dpkg-reconfigure exim4-config

ed utilizzare i seguenti parametri (in grassetto):

Tipo di configurazione del sistema di posta: posta inviata tramite uno «smarthost»; ricevuta via SMTP o fetchmail
Mail name del sistema: lfvm.test
Indirizzi IP sui quali attendere connessioni SMTP in ingresso: 127.0.0.1 ; ::1
Altre destinazioni per conto delle quali accettare posta: lfvm.test
Sistemi per i quali fare il «relay»: <vuoto>
Omettere il mail name locale dai messaggi in uscita? <No>
Mantenere al minimo il numero di richieste DNS (dial-on-demand)? <No>
Modalità di consegna per la posta locale: Formato mbox in /var/mail/
Dividere la configurazione in molti piccoli file? <No>

i parametri nome_smarthost e la porta_smarthost  sono quelli del proprio provider di posta elettronica, nel caso ad es. di Gmail:

Indirizzo IP o hostname per lo «smarthost» in uscita: smtp.gmail.com::587

Exim risulta quindi configurato per instradare i messaggi di posta elettronica attraverso un server SMTP esterno e  per completarne la configurazione occorre impostare il nome utente e la password che si devono utilizzare per l’accesso a tale servizio.

Dalla shell dei comandi dell’utente root con un editor aprire il file di configurazione /etc/exim4/passwd.client

root@lfvm:~# pico /etc/exim4/passwd.client

e modificare i parametri evidenziati in grassetto

# password file used when the local exim is authenticating to a remote
# host as a client.
#
# see exim4_passwd_client(5) for more documentation
#
# Example:
### target.mail.server.example:login:password
nome_smarthost:user:password

dove nome_smarthost è quello indicato nella precedente configurazione di Exim e user e password sono quelli da utilizzare per il login al servizio. Nel caso ad es. di Gmail risulta funzionante la seguente configurazione:

### target.mail.server.example:login:password
gmail-smtp.l.google.com:user:password
*.google.com:user:password
smtp.gmail.com:user:password

Per la verifica, eseguire il reload di Exim

root@lfvm:~# /etc/init.d/exim4 reload

inviare una mail di prova

root@lfvm:~# echo 'ok' | mail -s 'Exim4 - SMTP Relay Test' amministrazione.libera@gmail.com

e verificare nel log di Exim

root@lfvm:~# tail -f /var/log/exim4/mainlog

che non siano presenti errori

2013-11-26 17:48:49 1VlLoc-0001BO-DO => amministrazione.libera@gmail.com R=smarthost T=remote_smtp_smarthost H=gmail-smtp-msa.l.google.com [74.125.136.108] X=TLS1.2:RSA_ARCFOUR_SHA1:128 DN="C=US,ST=California,L=Mountain View,O=Google Inc,CN=smtp.gmail.com"

3) Tutorial per la configurazione base di LQFB

LQFB_VM è distribuita con LQFB perfettamente funzionante e configurato in maniera dimostrativa per permetterne un primo rapido utilizzo.

La piattaforma è ovviamente personalizzabile in ogni suo aspetto e secondo ogni esigenza che sia compatibile con i principi realizzativi della piattaforma, per un approfondimento si consiglia l’articolo: La democrazia interattiva con “Liquid Democracy”.

In questo breve tutorial si intendono riassumere i principi di funzionamento di LQFB  in funzione delle problematiche relative alla configurazione base ottenuta nella  VM.

LQFB nasce come sistema per creare, elaborare e sviluppare proposte, favorendo l’interazione e la comunicazione tra i suoi utenti. E’ basato sui seguenti 4 cardini:

  • Iniziativa: ogni utente può decidere di sottoporre, compilando un apposito form sulla piattaforma LQFB, una propria proposta.
  • Discussione: gli utenti discutono la mozione postando suggerimenti che l’autore della proposta  ha discrezione di implementare o meno; i membri possono anche fare proposte alternative in contrapposizione con quella inizialmente messa in discussione.
  • Voto: nel momento in cui l’iniziativa arriva alla votazione gli utenti possono partecipare decidendo quale iniziativa meriti il loro consenso, ma anche delegare (per la particolare iniziativa e/o per l’intera area di argomento) altri utenti a votare in loro vece. La delega è revocabile in qualsiasi momento.
  • Esecuzione: se l’iniziativa viene approvata (secondo il metodo matematico Shulze che seleziona come vincitrice la proposta mediamente più approvata) diventa esecutiva ed ha la forza e la voce dell’intera comunità di utenti.

Tutta la piattaforma, dalla struttura del database all’interfaccia web, si evolve e si concretizza intorno ai predenti principi.

Dal punto di vista formale si distinguono i seguenti elementi:

  • Proposta: le iniziative/soluzioni avanzate dagli utenti.
  • Tematica: insieme delle proposte relative a quel determinato problema/situazione/idea.
  • Aree: raccolgono le Tematiche relative ad un determinato argomento (Ad es.: agricoltura, organizzazione, economia, …)
  • Sezioni: sono contenitori di Aree che possono rappresentare zone territoriali (Ad es.: comuni, provincie, regioni) o Gruppi di Lavoro.

 

3.1) Configurazione del DB

La configurazione di LQFB si realizza in maggior parte nella fase iniziale operando direttamente sul suo database (per approfondimenti si consiglia la “Guida Database SQL”). Per la LQFB_VM tali operazioni possono essere svolte dall’utente www-data che ha i privilegi di creazione dei database in PostgreSQL.

Tutti gli script *.sql di inizializzazione e di supporto al DB sono disponibili nella cartella /opt/liquid_feedback_core/ si consiglia, per la struttura del DB, una consultazione del file core.sql.

L’utente www-data è senza password, pertanto si deve utilizzare l’utente root  per acquisirne i privilegi:

root@lfvm:~# su - www-data
$

per la cancellazione di un precedente DB

$ cd /opt/liquid_feedback_core
$ dropdb liquid_feedback
$

per l’inizializzazione di un nuovo DB

$ createdb liquid_feedback
$ createlang plpgsql liquid_feedback
$ psql -v ON_ERROR_STOP=1 -f core.sql liquid_feedback
$

per la connessione a PostgreSQL

$ psql liquid_feedback

psql (9.1.9)
Digita "help" per avere un aiuto.

liquid_feedback=>

per la disconnessione a  PostgreSQL

liquid_feedback=> \q
salvataggio della cronologia nel file "/var/www/.psql_history" fallita: Permesso negato
$

3.1.2) Definizione delle politiche disponibili per l’iter delle PROPOSTE

Ogni politica deve prevedere i tempi di durata delle varie fasi, i quorum necessari al passaggio alla fasi successive ed i rapporti di maggioranza necessari perché le proposte possano essere considerate vincitrici. Le proposte nel modello di LQFB sono sottoposte alle 4 fasi:

  • Nuovo: Una proposta appena creata rimane in questa fase per determinato periodo di tempo (stabilito dalla “Politica” scelta) al termine del quale se ha superato un prestabilito quorum di interesse (un certo numero di persone che cliccano per dimostrare il loro “interesse/sostegno” alla proposta) passa alla fase successiva;
  • Discussione: la/le Proposta/e sono quindi dibattute/discusse e possono essere modificate e/o aggiornate, tenendo anche conto dei suggerimenti o delle controproposte che ogni iscritto può fare: anche in questo caso superato un periodo di tempo prestabilito dalla Politica adottata, le proposte passano alla fase successiva;
  • Sospeso: in questa fase le proposte rimangono congelate per permettere di valutarle compiutamente e darne o meno l’appoggio necessario al passaggio successivo (quorum di interesse)
  • Votazione: Fase del voto, in cui ogni iscritto indipendentemente dall’interesse/coinvolgimento dimostrato nelle proposte può votarle.

3.2) L’interfaccia WEB di LQFB

All’indirizzo http://192.168.1.5/lf/admin/index.html l’amministratore di LQFB  può,  con maggiore comodità, creare e modificare:

  • Policy,
  • Sezioni,
  • Utenti;

inoltre, accedendo alla scheda dei singoli utenti, può decidere a quale Sezione quest’ultimi possono partecipare attivamente; lo stesso admin può inserirsi nella lista dei partecipanti ad una Sezione.
